Introduction

L'obiettivo principale dello sprint 0 è analizzare i requisiti forniti dal committente riguardo al progetto cargoservice, eliminando eventuali ambiguità relative ai termini utilizzati e formalizzando quest'ultimi con dei componenti software (che possono essere già sviluppati o ancora da sviluppare).

Una volta formalizzati i requisiti si procederà con la definizione di una prima architettura logica generale del sistema che sarà il riferimento iniziale per lo sprint 1.

Questa architettura iniziale sarà costituita dai macrocomponenti principali (bounded context) deducibili dal documento dei requisiti, e dalle interazioni tra quest'ultimi sotto forma di messaggi.

L'architettura logica sarà anche corredata di un relativo piano di test iniziale.

Ultimo obiettivo dello sprint 0 sarà quello di definire un piano di lavoro in cui si definirà il numero e gli obiettivi degli sprint necessari al completamento del sistema, decidendo come suddividere il lavoro.

Premessa | metamodello QAK

L'analisi dei requisiti riportata in questo sprint iniziale verrà eseguita basandosi sul concetto di attore e sfruttando il metamodello QAK e la relativa software factory.

L'utilizzo del concetto di attore permette di modellare le entità del sistema che hanno un proprio flusso di controllo (attive) e che interagiscono con altre entità mediante scambio di messaggi. Questa tipologia di entità è molto presente nel dominio; se si fossero utilizzati dei semplici POJO (come in UML) l'abstraction gap tra le entità da modellare e gli strumenti di modellazione sarebbe stato troppo alto.

Il metamodello QAK ci fornisce inoltre una software factory in grado di generare codice eseguibile (es. Kotlin) partendo dal modello, riducendo le fasi di implementazione successive. Tra i concetti fondamentali forniti da QAK, sfrutteremo in particolare i messaggi (scambiati tra attori con specifica sintassi e semantica) e i contesti (nodi logici in un sistema distribuito).

Requirements

Il testo completo dei requisiti fornito dal committente è disponibile al seguente link: Tema2026.html

The company asks us to build service named cargoservice that should work as follows.

The cargoservice is able to receive a request to load a container sent by some customer by using the pushbutton of the IOPort.

  • It sends the answer retrylater, if the IOPort is currently occupied by a container or if the system is Out of service
  • It rejects the request when the hold is already full, i.e. the slots1-4 are already occupied.
  • Otherwise, it considers the system as engaged, detects a free slot and returns as answer the name of such a reserved slot. While engaged, the system must blink a Led.

When the request to load is accepted, the customer must move the container in the sensor area within prefixed amount of time (e.g. 30 secs), otherwise the systems becomes disengaged. Then, the cargoservice uses the cargorobot to move the container from the IOPort to the slot5 (for marking the container) and then to the reserved slot.

The service must also show on the display on the IOPort:

  • the current state of the hold
  • the message 'Service working', when it is all is going well
  • the message 'Out of service' if the sonar sensor measures a distance D > DFREE for at least 3 secs (perhaps a failure of the sonar).

Differential Drive Robot (DDR) e Cargorobot

Un Differential Drive Robot (DDR) è un robot fisico o simulato che si muove grazie a ruote motrici indipendenti. All'interno del sistema, il DDR è il supporto fisico (fornito dal committente o tramite un WEnv) che viene comandato logicamente.

Il cargorobot è un robot logico capace, sotto richiesta, di muoversi liberamente nel deposito, recuperare un container dall'IO-port e posizionarlo in uno slot libero. Si tratta di un componente attivo con un proprio flusso di controllo, modellabile come attore. Sostanzialmente, fungerà da interfaccia logica (o driver) che maschera le specifiche del DDR fisico o virtuale sottostante, riducendo l'abstraction gap.

Stiva (hold) ed Elementi della Stiva

Il deposito/stiva (hold) rappresenta il workspace del cargorobot, di forma rettangolare. Contiene aree notevoli fisse:

  • slots1-4 (laydown-positions): Le postazioni finali in cui depositare i container.
  • slot5: La postazione per la marcatura, considerata ai fini del routing come un ostacolo permanente.
  • IOPort (pickup-position): La postazione di arrivo dei container dal cliente.
  • HOME: La postazione da cui il cargorobot parte e a cui fa ritorno.

Tali aree e il deposito stesso hanno caratteristiche passive. Tuttavia, per via del Single Responsibility Principle, piuttosto che un grande POJO aggregatore, è opportuno che il tracciamento dei container sia gestito internamente al cargoservice o, se distribuito, modellato con logiche separate per non accoppiare la gestione dei dati (stato stiva) con la logica di business pura.

Cargoservice, Sonar, Led e Marker

Il cargoservice è il macrocomponente principale, l'orchestratore del sistema. Decide se accettare (stato engaged) o rifiutare una richiesta (rispondendo retrylater), assegna gli slot (slots1-4) e coordina i movimenti del cargorobot. È un componente attivo e reattivo, modellabile nativamente come attore QAK.

Il Sonar è un sensore che rileva ciclicamente la distanza. Siccome può generare asincronamente un evento di allarme se misura una distanza D > DFREE prolungata, è un componente logico attivo che andrà modellato come attore.

Il Led è un attuatore che deve riflettere lo stato del sistema (lampeggiando se engaged o acceso fisso se fuori servizio). Pur essendo passivo (non decide da solo), siccome presumibilmente risiederà su un nodo fisico (Raspberry) remoto, verrà modellato anch'esso come attore distribuito.

Il Marker situato nello slot5 applica etichette (marcatura). Svolge un lavoro su comando e notifica la fine, configurandosi come attore eseguibile.

I dispositivi cliente (es. pushbutton e display sull'IOPort) sono interfacce esterne (Alien) che scambiano messaggi e stringhe (es. "Service working", "Out of service") con il cargoservice.

Test plans | Test di Sistema

Il test di sistema è un collaudo interno che ha il preciso compito di confermare il corretto funzionamento della rete e delle interazioni via messaggi tra i vari componenti (mock in questa prima fase). Questo obbliga a riflettere sulle interazioni prima di realizzarle.

Scenario 1: Richiesta accettata da cargoservice 

@Test
public void testLoadRequestAccepted() throws Exception {
    // Richiesta di carico
    String requestStr = CommUtils.buildRequest("tester", "load", "load(1)", "cargoservice").toString();
    System.out.println("Richiesta: " + requestStr);
    
    // Risposta accettata con slot assegnato
    String response = conn.request(requestStr);
    System.out.println("Risposta: " + response); 
    
    assertTrue("TEST: richiesta accettata", response.contains("load_accepted"));
}
Scenario 2: Richiesta rifiutata per mancanza di slot 

@Test
public void testLoadRequestDeniedFull() throws Exception {
    // Si suppone stiva gia' riempita...
    String requestStr = CommUtils.buildRequest("tester", "load", "load(1)", "cargoservice").toString();
    
    // Risposta rifiutata 
    String response = conn.request(requestStr);
    
    assertTrue("TEST: richiesta rifiutata", response.contains("load_refused") && response.contains("full"));
}
Scenario 3: Guasto Sonar e rifiuto (Out of service) 

@Test
public void testLoadRequestDeniedOutOfService() throws Exception {
    // Si emette un allarme sonar prolungato...
    // Poi si invia la richiesta
    String requestStr = CommUtils.buildRequest("tester", "load", "load(1)", "cargoservice").toString();
    String response = conn.request(requestStr);
    
    assertTrue("TEST: richiesta rifiutata per fuori servizio", response.contains("retrylater"));
}
