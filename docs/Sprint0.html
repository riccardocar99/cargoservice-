LABORATORIO DI INGEGNERIA DEI SISTEMI SOFTWARE

Introduction

Questo documento descrive il lavoro svolto nello Sprint 0 per lo sviluppo del sistema cargoservice. L'obiettivo principale è analizzare, formalizzare e delineare il modello concettuale del sistema software che automatizza le operazioni di carico dei container nella stiva di una nave utilizzando un robot a guida differenziale (cargorobot).

Requirements

Il testo completo dei requisiti fornito dal committente è disponibile al seguente link: Tema2026.html

The company asks us to build service named cargoservice that should work as follows.

The cargoservice is able to receive a request to load a container sent by some customer by using the pushbutton of the IOPort.

  • It sends the answer retrylater, if the IOPort is currently occupied by a container or if the system is Out of service
  • It rejects the request when the hold is already full, i.e. the slots1-4 are already occupied.
  • Otherwise, it considers the system as engaged, detects a free slot and returns as answer the name of such a reserved slot. While engaged, the system must blink a Led.

When the request to load is accepted, the customer must move the container in the sensor area within prefixed amount of time (e.g. 30 secs), otherwise the systems becomes disengaged. Then, the cargoservice uses the cargorobot to move the container from the IOPort to the slot5 (for marking the container) and then to the reserved slot.

The service must also show on the display on the IOPort:

  • the current state of the hold
  • the message 'Service working', when it is all is going well
  • the message 'Out of service' if the sonar sensor measures a distance D > DFREE for at least 3 secs (perhaps a failure of the sonar).

Requirement analysis

Sulla base dei requisiti forniti, costruiamo un primo dizionario per identificare e mappare i termini del dominio su concetti computazionali formali forniti dal metamodello QAK.

Identificazione dei Componenti e Dizionario (Nomi)

Termine del Dominio Natura / Entità Logica (QAK) Descrizione Formale
cargoservice QActor Attore (componente software attivo) che agisce come orchestratore centrale. Implementa la logica di business.
cargorobot QActor Attore che modella il robot a guida differenziale. Il sistema non distingue tra robot fisico e simulato (VirtualRobot).
IOPort Pushbutton Event Emitter (Alieno) Dispositivo di input hardware/software in grado di emettere eventi o richieste verso cargoservice.
IOPort Display QActor Dispositivo di output che mostra lo stato aggiornato del sistema, controllato da cargoservice.
Sonar Sensor Event Emitter / QActor Sensore di distanza. Emette eventi di dispatch al superamento di certe soglie (D > DFREE).
Marker QActor Dispositivo attivo nello slot5. Riceve il comando di marcatura ed emette una notifica di fine lavoro.
Led QActor Attuatore visivo di output. Riceve comandi per accendersi o lampeggiare a seconda dello stato di cargoservice.
hold (Stiva) Struttura Dati / POJO Modello logico interno allo stato del cargoservice, suddiviso in slots1-4 e slot5.

Stati del Sistema ed Azioni Logiche (Verbi)

Le azioni logiche sono formalizzate come macro-stati all'interno della FSM (Finite State Machine) dell'attore cargoservice:

  • Stato di attesa / validazione: ricezione della richiesta load, controllo di disponibilità (stato engaged) e risposta retrylater o assegnazione dello slot.
  • Stato di posizionamento: attesa temporizzata (30 secondi). Se scatta il timeout, ritorno a stato iniziale (disimpegno).
  • Sequenza logica coordinata: interazione con cargorobot per percorrere le tappe: IOPort → slot5 → slot assegnato → HOME.

Problem analysis

Chiariti i requisiti ed individuate le entità logiche, analizziamo i problemi architetturali legati alla costruzione del sistema. Poiché il sistema interagisce con diversi dispositivi hardware (e.g. Robot, Sonar, Marker, Led) e software in esecuzione in nodi distribuiti, adottiamo l'infrastruttura QAK per modellare nativamente un sistema distribuito a scambio di messaggi.

Modello delle Interazioni Logiche

Il coordinamento tra gli attori deve avvenire tramite un preciso scambio di messaggi, modellati secondo la semantica del linguaggio QAK:

  • Request / Reply (Interazione sincrona/asincrona bidirezionale): Utilizzata dal Cliente verso il cargoservice per la richiesta di scarico (request load : load(X) e reply load_ans : load_ans(SLOT)). Anche l'interazione con il robot per lo spostamento prevede una Response per notificare l'avvenuto arrivo a destinazione.
  • Dispatch (Messaggio asincrono "fire-and-forget" destinato ad un preciso ricevente): Utilizzato dal cargoservice per inviare comandi fire-and-forget al Led, al Display o al Marker.
  • Event (Messaggio emesso verso tutti gli interessati / publish-subscribe): Il Sonar emetterà eventi per notificare all'intero sistema (o al cargoservice in ascolto) le variazioni di distanza o anomalie prolungate (timeout > 3s).

Test plans

Al fine di verificare preventivamente che il modello logico progettato e il codice sviluppato rispettino i requisiti forniti dal committente, definiamo i seguenti piani di testing funzionale, che andranno automatizzati (e.g. tramite JUnit) per un corretto Continuous Testing.

Test Funzionale 1: Flusso Normale (Happy Path)
Obiettivo: Verificare il core-business a condizioni ideali.
Stato Iniziale: Stiva vuota. Robot in HOME. Sonar funzionante.
Azione: Si invia una Request load. Si attende l'assegnazione dello slot, si simula la presenza del container. Si verifica il corretto iter di dispacciamento del robot (HOME → IOPort → slot5 → slot assegnato → HOME).
Test Funzionale 2: Rifiuto Stiva Piena
Obiettivo: Assicurare che non venga mai superata la capienza massima (nessun over-booking).
Stato Iniziale: Gli slot 1-4 sono impostati come occupati.
Azione: Si invia una Request load. Si verifica che il sistema risponda con l'appropriato messaggio di rifiuto della richiesta.
Test Funzionale 3: Gestione Timeout Cliente
Obiettivo: Testare la robustezza temporale (timeout 30s) per l'inserimento del container.
Stato Iniziale: Stiva vuota. Robot in HOME.
Azione: Si invia una Request load. Il test non invia conferme di posizionamento del container per più di 30 secondi. Si verifica che il sistema torni allo stato disimpegnato liberando lo slot temporaneamente riservato.
Test Funzionale 4: Guasto al Sonar (Fuori Servizio)
Obiettivo: Testare la gestione dell'evento di errore inviato dal sonar (D > DFREE per 3s).
Azione: Si forza l'invio prolungato di eventi anomali dal sonar. Si verifica che il cargoservice transiti nello stato di "Out of service" e risponda con retrylater a qualsiasi nuova request load.
